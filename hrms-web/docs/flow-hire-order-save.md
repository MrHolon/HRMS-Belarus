# Что происходит при сохранении приказа о приёме (кнопка «Сохранить»)

Пошаговый поток данных при нажатии «Сохранить» в карточке пункта приказа о приёме (тип «Приём», `item_type_number = 1`).

---

## 1. Откуда берутся данные

В карточке пункта (OrderItemCard) для типа «Приём» пользователь заполняет:

- **Дата начала** → `effective_from` (дата приёма)
- **Должность** → в `payload` как `positionId` (uuid из `positions`)
- **Подразделение** → в `payload` как `departmentId` (uuid из `departments`)
- **Тип трудового документа** → в `payload` как `contractType` (`employment_contract` | `contract`)
- **Вид срока** → в `payload` как `contractTermKind` (например `indefinite`, `fixed`)
- **Срок действия договора** → в `payload` как `validFrom`, `validTo` (даты в ISO)
- Шаблон пункта, шаблон договора, дни отпуска и т.д. → тоже в `payload`

Из контекста приказа известны: `order_id`, `branch_id`, из пункта — `person_id` (сотрудник/кандидат).

---

## 2. Два сценария при нажатии «Сохранить»

### Сценарий A: Должность и/или подразделение не указаны

Тогда **только сохраняется черновик** пункта, приём не применяется.

| Шаг | Куда (таблица / API) | Действие |
|-----|----------------------|----------|
| 1 | **order_items** (update) | Обновляется строка пункта: `item_type_number`, `effective_from`, `effective_to`, `payload` (в т.ч. body, positionId, departmentId, …). |
| 2 | **candidates** (update) | Для этого `person_id` и `branch_id` статус кандидата ставится **prehire** (через фронт: get candidates → find by person_id/branch_id → update status). |

**В БД:** меняются только `order_items` (одна строка) и `candidates` (одна строка). Занятость и назначение **не создаются**, человек ещё не сотрудник.

---

### Сценарий B: Указаны должность, подразделение и дата начала

Тогда пункт **сохраняется и сразу применяется** — создаётся занятость и назначение, человек становится сотрудником.

Последовательность вызовов (всё через n8n CRUD):

| № | Таблица | Действие | Что записывается |
|---|---------|----------|-------------------|
| 1 | **order_items** | UPDATE | `item_type_number`, `effective_from`, `effective_to`, `payload` (текст, positionId, departmentId, contractType, contractTermKind, validFrom, validTo, contractTemplateId, leaveDays и т.д.). Пункт пока в статусе draft. |
| 2 | **employments** | INSERT | Новая строка занятости: `branch_id`, `person_id`, `employment_type = 'main'`, `status = 'active'`, `start_date` = дата из пункта, `hire_order_item_id` = id пункта приказа. Триггер БД подставляет `branch_id` при необходимости. |
| 3 | **assignments** | INSERT | Новая строка назначения: `branch_id`, `employment_id` (только что созданная), `department_id`, `position_id`, `start_date`, `basis_order_item_id` = id пункта приказа. Одна занятость → одно текущее назначение (должность/подразделение). |
| 4 | **contracts** | INSERT | Новая строка договора: `branch_id`, `person_id`, `employment_id` (созданная на шаге 2), `contract_type`, `contract_term_kind`, `valid_from`, `valid_to`, `hire_order_item_id` = id пункта приказа. Данные берутся из payload пункта (contractType, contractTermKind, validFrom, validTo). |
| 5 | **order_items** | UPDATE | Та же строка пункта обновляется: `state = 'applied'`, `applied_at` = текущее время, `employment_id` = id созданной занятости, `contract_id` = id созданной записи в contracts. |
| 6 | **candidates** | UPDATE | Для этого `person_id` и `branch_id` статус кандидата ставится **closed** (на фронте: get candidates → find by person_id/branch_id → update status). |

После шага 5 срабатывает триггер в БД **order_items_after_update_hire_sync_candidate**: при переходе пункта приёма в `state = 'applied'` он тоже выставляет кандидату `status = 'closed'`. То есть кандидат закрывается и с фронта (шаг 6), и при необходимости дублируется в БД триггером.

---

## 3. Сводка по таблицам при полном применении (сценарий B)

| Таблица | Что происходит |
|---------|-----------------|
| **order_items** | Два UPDATE: сначала payload/даты, потом state = applied, applied_at, employment_id, contract_id. |
| **employments** | Одна новая строка: активная занятость по филиалу, привязана к пункту приказа через `hire_order_item_id`. |
| **assignments** | Одна новая строка: должность и подразделение на дату приёма, привязка к занятости и к пункту через `basis_order_item_id`. |
| **contracts** | Одна новая строка: тип договора, вид срока, срок действия (valid_from, valid_to), привязка к занятости и к пункту через `hire_order_item_id`. В пункте приказа проставляется `contract_id`. |
| **candidates** | Один UPDATE: у записи по этому person_id и branch_id статус меняется на **closed**. |

Данные **не** попадают в:

- **absence_periods** — создаётся только при применении пунктов отпуска/командировки, не при приёме.
- **orders** — шапка приказа не меняется при сохранении пункта.

---

## 4. Связи после применения

- **persons** — не меняется (уже был создан при добавлении кандидата).
- **orders** — не меняется (сводный приказ уже выбран).
- **order_items** — пункт приёма: `employment_id` указывает на созданную занятость, `contract_id` — на созданный договор, `state = 'applied'`.
- **employments** — у новой записи: `hire_order_item_id` указывает на этот пункт, `person_id` — на человека.
- **assignments** — у новой записи: `employment_id`, `basis_order_item_id` (этот пункт), `department_id`, `position_id`.
- **contracts** — у новой записи: `employment_id`, `hire_order_item_id` (этот пункт), `contract_type`, `contract_term_kind`, `valid_from`, `valid_to`.
- **candidates** — одна запись по person_id/branch_id со статусом **closed**.

Так при сохранении приказа о приёме с заполненными должностью и подразделением данные расходятся в **order_items**, **employments**, **assignments** и **candidates**; визуально человек переходит из кандидатов в сотрудники (по представлению v_persons_list и логике раздела «Сотрудники»).

---

## 5. Смена типа пункта после применения приёма

Если пользователь **уже сохранил** пункт приказа о приёме (применён: созданы занятость и назначение), а затем меняет тип пункта на другой (например, «Прочий») и снова нажимает «Сохранить»:

- **Раньше:** в БД обновлялась только строка **order_items** (`item_type_number` и `payload`). Занятость и назначение **не удалялись** — человек оставался сотрудником, но пункт в приказе числился уже как «Прочий». Получалась несогласованность: занятость создана пунктом «Приём», а в приказе пункт отображался как «Прочий».
- **Сейчас:** для **применённого** пункта приёма (`state = 'applied'` и тип = Приём) выбор типа в карточке **заблокирован** (селект «Тип приказа» disabled). Подсказка (title) объясняет: тип применённого пункта приёма менять нельзя, занятость уже создана; чтобы отменить приём, нужно удалить пункт приказа (сработает триггер удаления и откат занятости/кандидата).

Таким образом, после применения приказа о приёме тип пункта нельзя сменить на «Прочий» или другой — только удалить пункт, если приём нужно отменить.

---

## 6. Изменение должности, подразделения, контракта и дней отпуска после применения

Если пункт приказа о приёме **уже применён**, а пользователь меняет в карточке должность, подразделение, шаблон контракта или дни отпуска и нажимает «Сохранить», происходит следующее.

### order_items (всегда)

- Выполняется **UPDATE** строки пункта: в **payload** (jsonb) записываются новые значения: `positionId`, `departmentId`, `contractTemplateId`, `leaveDays`, текст и т.д. То есть «документ» (текст приказа, подставляемые в шаблон данные) обновляется.

### Должность и подразделение (синхронизация с назначением)

- Раньше запись в **assignments** не обновлялась — фактическая должность/подразделение в системе оставались старыми, в приказе отображались новые (несогласованность).
- **Сейчас:** если пункт применён (`state = 'applied'`) и тип = Приём, при сохранении с новыми **positionId** и **departmentId** вызывается синхронизация: ищется назначение с `basis_order_item_id` = id этого пункта. Если назначение есть — выполняется **UPDATE assignments**: `position_id` и `department_id` из payload. Если назначения нет (например приём был применён без должности/отдела или из-за сбоя) — выполняется **INSERT** в assignments по данным пункта (employment_id, дата начала, должность, подразделение, basis_order_item_id). Таким образом, и приказ (payload), и фактическое назначение сотрудника (assignments) после сохранения совпадают. Показывается тост: «Пункт приказа сохранён, назначение обновлено».

### Контракт (тип, срок, даты) и шаблон печати

- **contract_type**, **contract_term_kind**, **valid_from**, **valid_to** при применении приёма записываются в таблицу **contracts** (шаги 4–5 выше). При последующем сохранении **уже применённого** пункта, если пользователь меняет тип документа, вид срока или даты действия, фронт выполняет **UPDATE contracts** по `contract_id` пункта (или по `hire_order_item_id`): обновляются поля `contract_type`, `contract_term_kind`, `valid_from`, `valid_to`. Таким образом, и приказ (payload), и запись договора в БД остаются согласованными.
- **contractTemplateId** и **leaveDays** хранятся только в **order_items.payload** и используются для печати/подстановки в шаблон. При изменении шаблона или дней отпуска обновляется только payload пункта — таблица **contracts** при этом не меняется.
