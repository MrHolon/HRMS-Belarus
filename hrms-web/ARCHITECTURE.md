# Архитектура HRMS Belarus (Frontend)

Фронт общается **только с n8n** (вебхуки). Supabase и другая логика — за n8n.

---

## Принципы

- **Один вход с фронта** — все запросы идут в n8n (вебхуки). Прямых вызовов Supabase/других БД с фронта нет.
- **Модульность** — один домен = одна зона ответственности (папка в `features`, маршруты в `app`).
- **Типы и контракты** — общие типы в `types/`, форматы запросов/ответов n8n — в `lib/n8n` или рядом.
- **Импорты** — через алиас `@/`; между доменами по минимуму, общее — через `lib` и `components`.

---

## n8n: как устроено

### Универсальный CRUD-вебхук

Один вебхук в n8n отвечает за **добавление, редактирование и удаление во всех таблицах**. Фронт передаёт:

- **таблица** (имя таблицы в системе);
- **действие** — `create` | `update` | `delete`;
- **поля** (тело: объект с полями для записи или идентификатор для удаления).

Формат и примеры задаются контрактом между фронтом и n8n (например, `{ table, action, id?, payload }`). n8n по `table` и `action` решает, куда писать (Supabase и т.д.) и что выполнять.

### Отдельные рабочие процессы

Специфичная логика выносится в **отдельные workflow’ы** в n8n:

- центр комплаенса (правила, проверки, каталог);
- согласования, уведомления, отчёты и т.п.

Для них — отдельные вебхуки/эндпоинты. Фронт вызывает их по имени процесса (например, «проверка по правилу X», «получить каталог правил»).

### Проверки при добавлении/редактировании

Зависит от ситуации:

- **Вариант A:** проверки встроены в CRUD-вебхук в n8n: при `create`/`update` n8n сам запускает проверки (в т.ч. комплаенс) и возвращает успех или ошибки валидации.
- **Вариант B:** отдельный вызов: фронт сначала вызывает вебхук проверки, при успехе — CRUD-вебхук.

Выбор за сценарием; архитектура допускает оба варианта.

## Центр комплаенса (законодательство РБ)

- **Вся логика и данные** — в n8n: правила, проверки, каталог правил, рабочие процессы центра комплаенса.
- **Фронт** — только UI: экраны раздела «Комплаенс», ввод параметров (при необходимости), кнопки типа «Проверить» / «Показать правило», отображение ответов n8n. Никакой своей логики правил и проверок.
- В коде центр комплаенса — обычный домен в `features/compliance` и маршруты в `app/(dashboard)/compliance/...`.

## Слой lib/n8n

Единая точка вызова n8n с фронта:

- **Базовый URL** вебхуков n8n (переменная окружения или константа).
- **Универсальный CRUD:** например `crud(table, action, payload)` — вызов одного вебхука с полями `table`, `action`, телом запроса.
- **Отдельные методы под процессы:** например `compliance.runCheck(...)`, `compliance.getRules(...)` — вызовы своих вебхуков n8n под центр комплаенса и другие workflow’ы.

Типы запросов/ответов (контракт с n8n) задать в `lib/n8n` или в `types/`, чтобы все вызовы были типизированы.

## Назначение слоёв

| Слой | Назначение |
|------|------------|
| **app/** | Маршруты, layout’ы, страницы. Минимум логики — композиция из `components` и `features`. |
| **app/(auth)** | Страницы входа/восстановления пароля. |
| **app/(dashboard)** | Все страницы после входа: дашборд, сотрудники, отпуска, документы, оргструктура, **комплаенс**, настройки. |
| **app/api** | При необходимости — приём входящих вебхуков от n8n (не исходящие вызовы с фронта). |
| **components/ui** | Только примитивы. |
| **components/layout** | Header, Sidebar, обёртки страниц. |
| **components/shared** | Таблицы, фильтры, карточки — без привязки к одному домену. |
| **features/*** | Доменные модули: компоненты, хуки, вызовы n8n через `lib/n8n`. Логика правил и данных — в n8n. |
| **features/compliance** | Центр комплаенса: UI и вызовы n8n; правил и проверок на фронте нет. |
| **lib/n8n** | Клиент к n8n: CRUD + методы под отдельные процессы. |
| **lib/constants** | Константы приложения (роли, статусы, коды). |
| **lib/validations** | Схемы для форм и входящих вебхуков (Zod и т.п.). |
| **types** | Общие типы (DTO, контракты с n8n). |
| **hooks** | Общие хуки (auth, theme, подписки на данные). |

## Именование

- **Компоненты:** PascalCase (`EmployeeCard.tsx`, `LeaveRequestForm.tsx`).
- **Хуки:** `use` + camelCase (`useEmployees.ts`, `useComplianceCheck.ts`).
- **Константы:** UPPER_SNAKE в `lib/constants`.

---

## Порядок развития

1. Задать контракт с n8n (формат CRUD, базовый URL) и реализовать `lib/n8n` (crud + при необходимости 1–2 метода под процессы).
2. Сделать `(auth)/login` и общий `(dashboard)/layout` (Header/Sidebar).
3. Добавлять домены по одному: сотрудники → отпуска → документы → оргструктура → центр комплаенса.
4. В каждом домене — только UI и вызовы `lib/n8n`; вся бизнес-логика и правила остаются в n8n.
